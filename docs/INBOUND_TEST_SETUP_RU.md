# Тестовая настройка почты: что нужно сделать владельцу

Мы настраиваем оба направления сразу, но безопасно:

- **IMAP** — получить ровно одно письмо в тестовый журнал `CAPTURED`; тикет, пост, вложение, автоответ и исходящее письмо не создаются.
- **SMTP** — сохранить реальные настройки отправки, но физическая отправка выключена до отдельного подтверждённого теста на один адрес.

Никогда не отправляйте пароль в чат, GitHub, тикет или скриншот. Если пароль уже попал
в чат, отзовите его у провайдера и создайте новый тестовый пароль приложения: старый считаем
скомпрометированным и не используем.

## 1. Возьмите параметры у почтового провайдера

В панели, где обслуживается `noc@23telecom...`, найдите раздел **Mail client setup**,
**IMAP/SMTP settings** или спросите администратора почты. Нужны ровно эти значения:

| Для получения (IMAP)             | Для отправки (SMTP)                          |
| -------------------------------- | -------------------------------------------- |
| сервер IMAP                      | сервер SMTP                                  |
| порт (обычно 993)                | порт (обычно 587, иногда 465)                |
| TLS включён/выключен             | 587 = STARTTLS (`false`), 465 = TLS (`true`) |
| полный адрес ящика               | логин SMTP (часто тот же полный адрес)       |
| пароль приложения / пароль ящика | пароль приложения / пароль SMTP              |

Если у ящика включена двухфакторная защита, создайте **пароль приложения** для почтового клиента. Не используйте личный пароль сотрудника.

Важно: SMTP сам по себе **не получает** письма. Для входящего теста обязательны IMAP-параметры.
Для IMAP заранее создайте **новую отдельную и пустую** папку/ярлык, например `Helpdesk Test`.
**Не используйте** общий INBOX, NOC Inbox, клиентскую папку, Gmail `All Mail` / Archive / Sent /
Drafts / Trash / Spam или их локализованные аналоги. Сервер дополнительно делает IMAP `LIST` и
откажется от папки с special-use флагом, невыбираемой папки либо папки, в которой уже есть хотя бы
одно письмо. Не используйте одну и ту же папку для двух canary-тестов: для нового теста создавайте
новую пустую папку. Имя этой папки внесём в поле `mailbox` только у выделенной тестовой очереди.
Важно: после первого запуска API в capture-only для этой очереди она получает необратимый маркер
`captureRetiredAt` ещё **до** подключения к IMAP. Даже если письмо не было принято или проверка
готовности не прошла, эту очередь и эту папку больше не используем для нового capture-теста.

## 2. Внесите значения локально

Откройте терминал на этой машине и выполните:

```bash
mkdir -p -m 700 ~/.config/23telecom-helpdesk
test -e ~/.config/23telecom-helpdesk/inbound-test.env || \
  cp docs/INBOUND_TEST_SECRETS_TEMPLATE.env ~/.config/23telecom-helpdesk/inbound-test.env
chmod 600 ~/.config/23telecom-helpdesk/inbound-test.env
nano ~/.config/23telecom-helpdesk/inbound-test.env
```

Заполните пустые значения в этом файле. В `TELECOM_HD_MAIL_FROM` укажите реальный адрес,
например `23 Telecom NOC <noc@23telecom.co.uk>` — не оставляйте `.example`.
В `TEST_IMAP_MAILBOX` укажите созданную выделенную папку/ярлык, а не `INBOX`.

Сохранить в `nano`: `Ctrl+O`, Enter, затем выйти: `Ctrl+X`.

До внесения чего-либо в приложение выполним безопасную проверку доступа к почте:

```bash
node scripts/verify-local-mail-credentials.mjs
```

Она только выполняет IMAP `LOGIN` + read-only `EXAMINE` выделенной папки и SMTP
`AUTH` по TLS, затем закрывает соединения. Скрипт **не** читает письмо, не меняет
папку и не отправляет письмо. Ожидаются две строки `IMAP: ... OK` и `SMTP: ... OK`.
Не вставляйте вывод с ошибкой вместе с какими-либо значениями файла; достаточно
сообщить, какой из двух этапов не прошёл.

## 3. Подготовьте локальную конфигурацию приложения

В корне репозитория выполните:

```bash
node scripts/provision-local-mail-test.mjs
```

Скрипт молча копирует только SMTP-параметры в игнорируемый `.env` и принудительно
оставляет получение, обработку и физическую отправку выключенными. IMAP-пароль не
копируется в `.env`: его позднее нужно внести через защищённую форму тестовой
`EmailQueue`, где он шифруется. Одновременно скрипт создаёт локальный
`TELECOM_HD_FIELD_ENCRYPTION_KEY` (или сохраняет уже корректный): он защищает пароль
IMAP в базе. Скрипт не печатает этот ключ; не удаляйте и не меняйте его во время теста.

После того как в `/admin/mail` создана выделенная IMAP-очередь и виден её числовой ID,
для подготовки **ровно одного** безопасного приёма используется только эта команда:

```bash
node scripts/set-local-imap-capture.mjs --queue-id 42
node scripts/preflight-capture-only.mjs
```

Замените `42` на ID выделенной очереди. Первая команда не запускает контейнеры и не
отправляет почту: она атомарно выставляет `outbound=false`, normal inbound=false,
`capture-only=true`, `IMAP=true`, `FROM_NOW`, `BACKFILL=0` и лимит `CAPTURE_MAX_MESSAGES=1`.
Вторая команда проверяет именно эти условия и ничего не меняет. После одного тестового
письма безопасное закрытие выполняется так:

```bash
node scripts/set-local-imap-capture.mjs --disable
docker compose up -d --no-build --force-recreate api
docker compose exec -T api wget -qO- http://127.0.0.1:4000/api/health
```

`--disable` закрывает глобальные гейты, но не сбрасывает capture-маркер. После команды обязательно
перезапустите API и дождитесь успешного health-ответа: только тогда считаем входящую тестовую
приёмку закрытой. Retired-очередь сохраняется как evidence для просмотра/строго ограниченного
promotion, но не может быть снова включена, удалена или использована для свежего IMAP/PIPE приёма.

## 4. Точный порядок одного входящего теста

1. В `/admin/mail` создайте одну **IMAP**-очередь с реальными IMAP-параметрами и папкой
   `Helpdesk Test`. Папка должна быть новой и пустой. Очередь может быть включена, но пока все
   глобальные гейты закрыты, приложение к почте не подключится. Не выбирайте `INBOX`, All Mail,
   Archive или системную папку.
2. Запишите её числовой ID в команду `set-local-imap-capture` из раздела 3, затем успешно
   выполните `preflight-capture-only.mjs`.
3. Перезапустите только локальный API с уже проверенным `.env`:

   ```bash
   docker compose up -d --no-build --force-recreate api
   ```

4. В `/admin/mail` дождитесь, пока именно эта очередь покажет `syncState=OK`, `uidValidity`,
   начальный `lastSeenUid` **и** health `captureTarget.ready=true`. Последнее означает, что
   работающий API заново подтвердил через IMAP `LIST`, что это выделенная не-special-use папка,
   и под mailbox lock увидел `exists=0`. Если UI показывает любой другой статус или alert
   `capture_target_not_ready`, не отправляйте письмо: закройте гейты, создайте **новую** очередь
   и новую пустую папку, затем начните заново. Уже armed queue не переиспользуется. Это граница
   `FROM_NOW`: письмо, лежавшее в папке раньше этой точки, намеренно не будет принято.
5. Только теперь вручную положите **одно** новое тестовое письмо в папку `Helpdesk Test`.
   Дождитесь не более двух 60-секундных циклов и проверьте: `CAPTURED=1`, нет нового тикета,
   поста, вложения, автоответа или исходящей команды.
6. Сразу закройте гейты командой `--disable` и повторите тот же перезапуск API. Не оставляйте
   capture-only включённым между тестами. После любой capture-only попытки — успешной, пустой или
   остановленной из-за readiness — следующий тест проводите только с новой очередью и новой пустой
   папкой.

SMTP уже сохранён, но физическая отправка всё это время остаётся выключенной. Отдельный
SMTP-тест на одного согласованного получателя возможен только после успешной проверки
входящего сценария и отдельного явного решения включить outbound gate.

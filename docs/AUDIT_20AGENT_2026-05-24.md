## TL;DR

The build is **genuinely green** (verified independently: typecheck/lint/build clean, vitest **556/556**, verify gate **9/9**, Playwright e2e **37/37** against a live stack). The architecture is competent and shows real security thinking.

**But it is NOT production-ready.** There are **3 confirmed critical defects** (one privilege escalation, one data-loss path, one runaway escalation loop), several high-severity security gaps, a dead mail queue that makes all email synchronous, unbounded queries that OOM at scale, and an entire GO_LIVE operational layer (health endpoint, backups, bootstrap admin, log rotation) that was never built. Docs are ~55% accurate. It is **pilot-ready, not go-live-ready** — which matches the project's own GO_LIVE.md (all items still open checkboxes).

## HIGH

**Security**

- **Refresh-token reuse not detected** — stolen refresh token = silent persistent access; no token-family revocation. `auth/auth.service.ts:87-110`
- **Throttler is in-memory** — login rate-limit (5/60s) is per-process; bypassable behind a load balancer / multi-instance. `app.module.ts:49`
- **Stored XSS** — KB article `body` rendered via `dangerouslySetInnerHTML` with no sanitization (no DOMPurify/sanitize-html anywhere). `web/.../kb/[slug]/kb-article-content.tsx:73`
- **`isEncrypted` custom fields stored in plaintext** — flag exists, encryption utils exist, but write paths never call them. Contradicts ADR-0002. `admin/admin.service.ts:80-94`
- **Agents hold `user.manage` + `org.manage`** — any agent can disable any customer, delete orgs (silently orphaning users), re-file users. `auth/permissions.ts:44-57`
- **Containers run as root**, **Swagger exposed in prod**, **prod Redis unauthenticated**. `apps/*/Dockerfile`, `api/src/main.ts:50`, `docker-compose.prod.yml`
- **PII leak** — `GET /tickets/my` (public, 30/60s) uses `include` not `select`, returning `ipAddress`/`customFields` for any email. `tickets/tickets.service.ts:275-305`
- **File upload** — `application/octet-stream` in allowlist + no magic-byte check + ClamAV is a TODO → arbitrary files stored. `attachments/attachments.controller.ts:44`

**Email / Load**

- **Mail BullMQ queue is dead code** — `MailProcessor` registered but `queue.add()` never called; all sends are synchronous → SMTP latency blocks HTTP threads & processor loops. `mail.processor.ts`, `tickets.service.ts:253,716`
- **SMTP credentials never read** — `SMTP_USER/PASSWORD` absent from config schema & transport → outbound mail fails on any authenticated relay. `mail/mail.service.ts:36-41`
- **Workflow executor has no loop guard** — a `status eq X → change_status to Y` rule on `ticket.status_changed` recurses. `workflow/workflow.executor.ts`
- **ILIKE search, no GIN/trgm index** — every ticket search = 2× seq-scan of full table. `tickets.service.ts:469-529`
- **`addWorkingSeconds` steps one minute at a time** — O(N) CPU per ticket on every create/reply/bulk; blocks event loop. `sla/sla.service.ts:76-96`
- **No Prisma connection-pool tuning** — API + 4 BullMQ workers share an untuned default pool → starvation. `prisma/prisma.service.ts`

**Migration**

- **5 of 9 carriers misclassified** — `SUPPLIER_NAMES` list misses KOLTELECOM, Text2World, Text2reach, PaaSoo, Broadband Group → they import as CLIENT, breaking the core client↔supplier model. `migration/kayako-parser.ts:158`
- **Ticket history not migrated** — 2,663 tickets / 6,029 posts out of scope; demo data only.

**Frontend completeness**

- **Admin:** Alaris page is a fully dead stub; custom-field **edit impossible** (no update hook); SLA schedule **workHours silently discarded** (SLA calc then treats all time as off-hours); staff disable & group management absent. `web/app/(admin)/...`
- **Staff:** silent attachment drop on internal notes; shared `register('body')` across reply/note tabs; kanban drag has no error rollback & no touch support; **merge/split/watcher APIs have no UI**. `web/app/(staff)/...`

**Docs**

- `endpoints.md` covers ~55% of routes; 6 controllers (attachments, follow-ups, time-tracking, saved-views, email-queues, parser-rules) entirely undocumented; `architecture.md` falsely lists attachment upload as a TODO; `database.md` missing 6 models + 4 enums.

## What's genuinely GOOD (verified)

- Build/test claims **hold**: 556/556 vitest, 9/9 verify gate, 37/37 e2e — all run live, no skips. (Docs say "547" — stale undercount, harmless.)
- Auth core: argon2id, access+refresh rotation, HttpOnly cookies, jti blocklist (observable fail-open), `assertProductionSecrets` boot guard, CORS locked to public URL.
- Path-traversal guard, orphan-claim-token IDOR prevention, saved-view per-user scoping — all correct and tested.
- Alaris webhook: constant-time secret compare, dedup, prod secret enforced at boot — works end-to-end (ADR-0005 mislabels it a "stub").
- Reports: no raw SQL injection (whitelist + Zod re-parse), CSV escaping correct, cron UTC-pinned.
- Client portal: ~95% wired (submit/view/reply/KB all functional with loading+error states).
- Spec fidelity for H1–H8 and M0–M4 is **high** — most claims back up to real code (one real overclaim: macro `replyText` textarea was never added to the UI).

## Commander's recommendation

1. **Block production.** Fix C1–C4 first (privilege escalation, merge data-loss, SLA loop, unbounded queries) — these are correctness/safety, not polish.
2. **Then the security HIGH set** (refresh-reuse, distributed throttler, KB XSS, encrypted-field plaintext, root containers, Swagger-in-prod, PII leak).
3. **Then load** (wire the mail queue or make sends async, add search indexes + pool tuning, replace minute-stepper SLA calc).
4. **Migration:** fix carrier classification before importing the real dump; plan ticket-history migration.
5. **Frontend:** finish admin (custom-field edit, SLA workHours, Alaris or remove it), staff merge/split/watcher UI, fix internal-note attachment drop.
6. **Docs:** regenerate endpoints/database/architecture from code — they cannot be trusted for onboarding or security review as-is.
7. Realistic capacity today: **~20–50 concurrent req/s**, not the unvalidated 1000 RPS k6 target.
